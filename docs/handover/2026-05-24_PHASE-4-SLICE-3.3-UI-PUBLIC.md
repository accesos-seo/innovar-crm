## Decisiones Técnicas Clave

### 1. **BankDetailsCard silenciosa vs. placeholder**
- **Motivo**: Evitar flash de "sin datos" si admin no configuró nada
- **Implementación**: `null` render si todos los campos están `undefined`/empty
- **UX**: Cliente ve solo lo que importa, no secciones vacías

### 2. **Copy-to-Clipboard con visual feedback**
- **Motivo**: Sin feedback, usuario no sabe si copió
- **Implementación**: State `copiedKey` → Check icon x 2s → vuelve a Copy icon
- **Alternativa considerada**: Toast (demasiado ruido para acción de bajo riesgo)

### 3. **PaymentProofUploader con Select restringido**
- **Motivo**: Cliente solo puede pagar digitalmente (transferencia/nequi/daviplata/pse)
- **Implementación**: `clientFacingPaymentMethodSchema.options` (4 métodos únicamente)
- **DB**: `advance/installment/final/refund` en inglés, UI siempre `PAYMENT_METHOD_LABELS_ES`

### 4. **Validation warning vs. error**
- **Motivo**: Monto bajo no es fatal, pero usuario debe enterarse
- **Implementación**: Warning amber si `amount < suggestedMin`, botón sigue activo
- **Criterio**: Error (botón desactiva) solo si file>5MB, invalid mime, amount=0/null, method=empty

### 5. **`useSetting<T>('bank_block')` con Record<string, any>**
- **Motivo**: Evitar query key explosion (sería `.in('key', KEYS)` con 7 keys = 7 queries)
- **Implementación**: Una key única `bank_block` que devuelve objeto plano `{ bank_name, bank_account_number, ... }`
- **⚠️ TODO en migraciones S3**: Confirmar que `system_settings` tiene fila con `key='bank_block'` que agrega/actualiza esta columna en `system_settings.value` como JSONB

### 6. **Feature flag OFF → zero breaking changes**
- **Motivo**: Seguridad producción, poder rollback sin impacto
- **Implementación**: Ternario inline en PublicQuotation, no guards globales
- **Fallback**: `PendingPaymentNotice` legacy sigue funcionando, usuarios no ven cambio

### 7. **Null coalescing para settings con defaults**
- **Motivo**: `useSetting` devuelve `T | null`, necesito `number` para `suggestedMinPct`
- **Implementación**: `typeof suggestedMinPct === 'number' ? suggestedMinPct : 30`
- **Alternativa rechazada**: `?? 30` (no typesafe, `undefined ?? 30` falla)

## Flujos Validados

### ✅ Flujo 1: Flag OFF
1. Usuario abre `/c/<code>` con cotización `client_approved`
2. `slice3 = false`
3. Renderiza `<PendingPaymentNotice />` (original)
4. UI idéntica a antes

### ✅ Flujo 2: Flag ON, datos banco configurados
1. Usuario abre `/c/<code>` con cotización `client_approved`
2. `slice3 = true`
3. Fetch `bank_block` y `suggested_min_advance_pct` → ok
4. Renderiza `<BankDetailsCard />` con 3 secciones (transfer + nequi + daviplata)
5. Renderiza `<PaymentProofUploader />` con 4 métodos
6. Usuario sube comprobante → mutation → RPC `submit_quotation_payment_proof` → status → `pending_payment_verification`
7. UI cambia a `<UnderReviewNotice />`

### ✅ Flujo 3: Flag ON, datos banco vacío
1. Usuario abre `/c/<code>` con cotización `client_approved`
2. `slice3 = true`
3. Fetch `bank_block` → null/empty record
4. `<BankDetailsCard />` renderiza null (sin secciones visibles)
5. Pero `<PaymentProofUploader />` sigue visible (usuario puede subir sin saber dónde pagar → **⚠️ UX issue**)
   - **Mitigación**: BankDetailsCard solo es null si TODOS los campos son vacíos. Admin debe llenar al menos uno.

### ✅ Flujo 4: Status transitions
- `pending_payment_verification` → muestra `<UnderReviewNotice />` (azul, Clock icon)
- `approved` → muestra `<ApprovedNotice />` (verde, Sparkles)
- `cancelled`, `superseded`, `expired`, `rejected` → renderizan sus propias vistas (no cambian con S3.3)
