## 🔴 Hotfix bloqueante: migración 040 (RLS anon de system_settings)

### Bug descubierto en este smoke

`PublicQuotation.tsx` usa:
- `useFeatureFlag('slice_3_enabled')` — gatea entre flujo legacy y S3.
- `useSetting<BankDetails>('bank_block')` — datos bancarios.
- `useSetting<number>('suggested_min_advance_pct')` — UX warning.

Los 3 hooks llaman `from('system_settings').select(...)` desde el cliente JWT `anon`. Pero la RLS pre-040 solo permitía SELECT a `authenticated`:

```sql
-- Antes:
system_settings_select  TO authenticated  FOR SELECT  USING (true)
```

Resultado: el cliente anon recibía `data: null` → `useFeatureFlag` caía al default `false` → siempre renderizaba `PendingPaymentNotice` legacy. **Cliente NUNCA vería el flujo S3 incluso con flag ON en DB**. Bloqueante de go-live S3.

### Fix aplicado

[db/migrations/040_fix_anon_read_public_system_settings.sql](../../db/migrations/040_fix_anon_read_public_system_settings.sql) agrega policy paralela `system_settings_select_public` para `anon` con allowlist de 10 keys safe-to-public:

```sql
CREATE POLICY system_settings_select_public ON public.system_settings
  FOR SELECT TO anon
  USING (key = ANY (ARRAY[
    'slice_3_enabled',
    'bank_name','bank_account_number','bank_account_type','bank_holder_name','bank_holder_id','nequi_phone','daviplata_phone',
    'suggested_min_advance_pct',
    'public_app_base_url'
  ]::text[]));
```

NO expone tokens Meta, flags internas ni claves administrativas. Aplicada en prod `xdzbjptozeqcbnaqhtye`. Idempotente (DROP IF EXISTS + CREATE).

## 🟠 Bug menor remanente: `bank_block` consolidated key

El frontend (`PublicQuotation.tsx` línea 27-28 y `BankDetailsCard`) lee:

```tsx
const { data: bankDetails } = useSetting<BankDetails>('bank_block');
```

Pero `system_settings` tiene 7 keys individuales (`bank_name`, `bank_account_number`, etc.), NO una key consolidada `bank_block`. Resultado: post-040 el componente recibe `{}` y renderiza `null` (silent card).

Esto es **el bug que el handoff S3.3 ya marcó** (§6 "TODO en migraciones S3: Confirmar que system_settings tiene fila con key='bank_block'..."). No se arregla en S3.5 por scope creep — es un refactor entre 2 opciones:

- **Opción A**: cambiar el hook a `useSetting<string>('bank_name')`... × 7 + agregar consolidación client-side. PRO: sin tocar DB. CON: 7 hooks.
- **Opción B**: crear migración que inserte `bank_block` como JSONB consolidado en `system_settings`, con trigger que se actualice cada vez que cambia una de las 7 keys individuales (o reemplazar las 7 por la consolidada). PRO: 1 hook. CON: refactor DB + admin page de BankSettings.

Estado real durante smoke: el `PaymentProofUploader` aparece, el cliente PUEDE subir comprobante (y la RPC `submit_quotation_payment_proof` funcionará). Solo falta que vea los datos bancarios in-page (los recibe igual por WhatsApp legacy).

**Recomendación**: priorizar este fix en S3.6 o como PR independiente antes del go-live S3.
