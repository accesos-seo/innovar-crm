## 🎯 Objetivo de la fase

Antes de Fase 1, el sistema tenía 3 problemas estructurales que **bloqueaban un lanzamiento real a producción**:

1. **Inseguro**: cualquier URL era accesible sin login (RLS era la única protección, frágil si se desactivaba accidentalmente).
2. **Mentiroso**: 85 sitios en el código retornaban `[]` ante errores de Supabase, mostrando toasts de éxito aunque la operación había fallado.
3. **No resiliente**: un error no manejado tumbaba la app entera con pantalla blanca. Sin diferenciación entre errores de red, RLS, sesión expirada o validación.

Adicionalmente había 2 bugs de seguridad sutiles:
- El `authStore` y el trigger `handle_new_user` asignaban rol `admin` a cualquier usuario nuevo.
- La generación de `quotation_number` tenía race condition bajo concurrencia.

## ✅ Lo que se hizo

### Capa de utilidades nueva

**Archivo creado**: `src/lib/errors.ts` (188 líneas)

Sistema central de manejo de errores con:
- `AppError` — clase con `code`, `message` (español), `pgCode` (Postgres), `userFacing` flag
- `mapSupabaseError(error)` — traduce cualquier error a `AppError` con mensaje humano
- `notifyError(error, title)` — muestra toast (con guard para errores no-user-facing)
- `assertSupabase(supabase)` — guard que lanza si el cliente es null
- `unwrapSupabase(response)` — atajo para `if(error)throw...; return data`

Tabla de mapeo Postgres → mensaje:
```
PGRST301 (JWT expired)     → "Tu sesión expiró"
42501    (insufficient_privilege) → "No tienes permisos"
PGRST116 (no rows)         → "El registro no existe"
23505    (unique violation)→ "Ya existe un registro"
23503    (foreign key)     → "Hay relaciones que lo impiden"
23502/23514 (not null/check)→ "Faltan campos requeridos"
Timeouts                   → "La operación tardó demasiado"
Network                    → "Sin conexión al servidor"
```

### Fix race condition: `quotation_number`

**Problema descubierto durante la fase**: la columna `quotation_number` **no existía** en la tabla `quotations` — era otro de los 85 errores silenciados. El frontend la usaba pero el DB nunca la registraba.

**Solución**: migración SQL `db/migrations/001_generate_quotation_number.sql` que:
1. Crea la columna `quotation_number TEXT` (si no existe)
2. Backfill de cotizaciones existentes con `COT-{year}-{padded_seq}`
3. Agrega constraint UNIQUE
4. Crea función `generate_next_quotation_number()` con `pg_advisory_xact_lock` (serializa concurrencia)
5. Otorga permisos a `authenticated` y `anon`

**Cliente**: `useCreateQuotation` ahora:
1. Llama al RPC `generate_next_quotation_number()` primero
2. Si falla (función no desplegada), cae a generación cliente
3. Detecta error `23505` (unique violation) y reintenta con siguiente número
4. Máximo 5 intentos antes de rendirse

Schema Zod (`src/schemas/quotation.ts`) actualizado para incluir `quotation_number` como campo opcional (lo asigna el RPC).

## 🧠 Lecciones aprendidas

### Lo que reveló el refactor

Al cambiar `return []` por `throw`, salieron a la luz **bugs reales** que estaban ocultos:

1. **La columna `quotation_number` no existía**. El frontend pretendía usarla pero el insert silenciosamente la perdía. Esto venía pasando potencialmente por meses.
2. **Errores de RLS aparecían como "guardado correctamente"**. Ahora salta el toast rojo correcto.
3. **Errores de red sin internet daban pantalla blanca**. Ahora se ve `<ErrorBoundary>` con botón "Reintentar".

> 🎓 **Aprendizaje**: silenciar errores no los elimina, solo los pospone. Es preferible un toast rojo claro hoy que un cliente confundido mañana.

### Patrones que NO funcionaron

- **Intentar arreglar 85 sitios manualmente** → muy lento y propenso a inconsistencias. Solución: crear helpers (`mapSupabaseError`, `assertSupabase`, `notifyError`) y aplicar el patrón mecánicamente.
- **Generar `quotation_number` en cliente con retry** sin función Postgres → todavía hay ventana de race. Solución correcta: advisory lock en función `SECURITY DEFINER`.

### Patrones que funcionaron muy bien

- **Helpers pequeños y composables** (`assertSupabase`, `mapSupabaseError`, `notifyError`) → cubren 95% de los casos en 4 líneas.
- **Documentar el "por qué" en docstrings** del archivo `errors.ts` → la próxima persona entiende la decisión sin arqueología de git.
- **Migración SQL idempotente** (con `IF NOT EXISTS`, `DO $$` blocks) → segura de re-ejecutar.
