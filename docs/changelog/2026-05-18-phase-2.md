## 🧠 Lecciones aprendidas

### Lo que funcionó bien

- **Schemas Zod por entidad** (un archivo por tabla) escala mejor que un mega-archivo. La organización quedó muy limpia.
- **Tipos manual + comando para regenerar** es buen compromiso entre control y mantenibilidad. El comentario al inicio del archivo deja claro qué hacer cuando cambie el schema.
- **Migraciones SQL idempotentes** (`IF NOT EXISTS`, `DROP IF EXISTS`, `DO $$`) son oro: seguras de re-ejecutar incluso si dudas.

### Lo que dolió

- **OneDrive + Windows** son un combo brutal para operaciones de filesystem masivas. `npx tsc --noEmit` se vuelve interminable. Recomendación a futuro: trabajar fuera de OneDrive si es posible (mover el repo a `C:\Dev\` por ejemplo).
- **Mi entorno bash restringido** (sin `head`, `find`, `node`) me hizo perder tiempo. Para tareas tipo "leer las primeras N líneas de un log" tuve que pivotar a PowerShell varias veces.
- **Quedé encerrado esperando el output de `tsc`** en lugar de seguir avanzando con otras tareas en paralelo. Aprendizaje para próxima sesión: si una tarea bloquea por entorno, mover a otra y volver.

### Patrón que vale repetir

Cuando un AppError nuevo aparezca (ej: necesitas un código `RATE_LIMITED` o `MAINTENANCE`), simplemente:
1. Agrégalo al tipo `AppErrorCode` en `src/lib/errors.ts`
2. Agrega su mapeo en `mapSupabaseError()`
3. Todo el resto del sistema ya lo entiende automáticamente.
